# 前端集成测试 - 游戏生成案例

本文档用于手动测试 Yaha 游戏生成平台的核心流程。每个案例列出**前端填写内容**、**期望的 Agent 执行路径**，以及**数据库和前端期望结果**。

---

## 测试环境准备

**API 调试终端**（agent-service）：

```bash
cd services/agent-service
uv run uvicorn app.main:app --host 127.0.0.1 --port 8000 --log-level debug
```

**前端**：
- 访问 `http://localhost:3000`
- 登录后进入"创建游戏"页面
- 填写"游戏名称"和"创意描述"（对应 API 的 `title` 和 `prompt`）
- 点击"创建生成任务"按钮

### 案例 4：简单游戏 - 点击得分（统一 LLM 生成路径）

**前端填写：**

| 字段 | 内容 |
|------|------|
| 游戏名称 | `点击得分小游戏` |
| 创意描述 | `做一个点击得分的游戏` |
| 素材 | 无 |

**期望 Agent 执行路径：**

```
START
  → SupervisorAgent → [route: "approved"]
    → specialist_fan_out（分发到 3 个 Specialist）
      → vision_agent（并行 LLM）
      → gameplay_agent（并行 LLM）
      → narrative_agent（并行 LLM）
    → synthesis_agent（LLM 整合设计）
      → code_generator（LLM 生成完整代码）
        → validator（验证产物结构）
          → upload_workflow（上传 OSS）
            → END
```

**期望日志（终端）：**

```
SupervisorAgent: 决策结果 status=approved, complexity=simple, task_id=xxx
SpecialistFanOut: 准备并行分发, task_id=xxx
VisionAgent: 生成视觉规范, task_id=xxx
GameplayAgent: 生成游戏机制, task_id=xxx
NarrativeAgent: 生成叙事内容, task_id=xxx
SynthesisAgent: 整合设计规范, task_id=xxx
CodeGenerator: 生成游戏代码, task_id=xxx
Validator: 验证生成代码, task_id=xxx
UploadWorkflow: 上传文件, task_id=xxx
```

**期望数据库 `GenerationTask` 记录：**

```json
{
  "status": "COMPLETED",
  "currentStep": "upload",
  "resultGameId": "game_xxx",
  "resultVersionId": "version_xxx",
  "errorMessage": null
}
```

**期望数据库 `AgentLog` 记录（至少 10 条）：**

| agent | step | message |
|-------|------|---------|
| SupervisorAgent | start | 开始分析用户输入 |
| SupervisorAgent | decision | 决策完成: approved, 复杂度: simple |
| SpecialistFanOut | prepare | 准备并行执行 Specialist Agents |
| VisionAgent | start/complete | 开始/完成视觉规范生成 |
| GameplayAgent | start/complete | 开始/完成游戏机制生成 |
| NarrativeAgent | start/complete | 开始/完成叙事内容生成 |
| SynthesisAgent | start/complete | 开始/完成设计整合 |
| CodeGenerator | start/llm_complete/validate_complete | 代码生成各阶段 |
| Validator | validate | 产物结构、manifest 与安全规则校验通过 |
| UploadWorkflow | upload_complete | 产物已上传对象存储 |

**期望前端显示：**

- 成功提示：`游戏"点击得分小游戏"已生成！`
- 显示生成的产物链接（manifest URL、入口 URL）
- 跳转到游戏预览页

### 案例 14：复杂游戏 - Validator 重试（验证失败 → 重试）

**前置条件：**

- 需要 `code_generator` 渲染的产物第一次验证不通过
- 触发 `should_retry` → `"retry_workflow"` → 修复 → `validator` 再次验证
- 此场景较难手动触发，通常由产物生成 bug 引发

**期望 Agent 执行路径：**

```
... → validator（第一次验证失败）
  → [should_retry 返回 "retry_workflow"]
    → retry_workflow（修复产物）
      → validator（第二次验证通过）
        → upload_workflow
          → END
```

**期望日志（终端）：**

```
Validator: 验证失败，产物结构异常
RetryWorkflow: 开始修复...
RetryWorkflow: 修复完成
Validator: 第二次验证通过
```

**期望数据库 `AgentLog`：**

| agent | step | message |
|-------|------|---------|
| Validator | validate | 验证失败 |
| RetryWorkflow | retry | 开始修复... |
| RetryWorkflow | retry_complete | 修复完成 |
| Validator | validate | 验证通过 |

**期望数据库 `GenerationTask`：`status = COMPLETED`（即使重试后仍成功）**

